## core components

| component | description | runs on |
|-----------|-------------|---------|
| **message adapter** | receives lark / slack webhook events, parses user commands, outputs unified `Command` objects | server |
| **overlord web** | browser-based dashboard вҖ” task management, live pty terminal, machine monitoring, admin panel | server |
| **task dispatcher** | scheduling engine вҖ” manages task queue (bullmq), selects workers, tracks task lifecycle, persists to sqlite | server |
| **worker agent** | execution engine вҖ” manages git worktrees, spawns pty terminals, runs ai agents, manages cursor tunnels | each worker machine |
| **pty manager** | worker sub-component вҖ” creates pseudo-terminals via node-pty, streams i/o bidirectionally | worker |
| **pipeline runner** | worker sub-component вҖ” monitors pty output, detects stage completion, injects next skill command | worker |
| **notifier** | sends notifications via the source platform (lark cards, slack blocks) and in-app notifications | server |
| **developer cli (ov)** | command-line tool вҖ” task creation, pty attach, project/machine queries, notifications | developer machine |

## data flow

1. **task creation** вҖ” developer creates a task via web, cli, or bot
2. **dispatching** вҖ” dispatcher selects the best available worker based on capacity, capabilities, and load
3. **workspace setup** вҖ” worker creates an isolated git worktree for the task
4. **execution** вҖ” pipeline runner drives the ai agent through configured stages
5. **monitoring** вҖ” pty output streams in real time to web dashboard and cli
6. **completion** вҖ” agent commits code, pushes branch, creates mr/pr
7. **notification** вҖ” notifier informs the developer through their original channel

## task state machine

```
QUEUED вҶ’ ASSIGNED вҶ’ RUNNING в”Җв”ҖвҶ’ COMPLETED
                       в”Ӯ  вҶҳ
                       в”Ӯ   CANCELLED
                       вҶ“       вҶ‘
                   SUSPENDED в”Җв”Җв”Ө
                       в”Ӯ    в”Җв”ҖвҶ’ RUNNING (reconnect)
                       в”Ӯ    в”Җв”ҖвҶ’ COMPLETED
                       в”Ӯ    в”Җв”ҖвҶ’ FAILED (timeout)
                       в”Ӯ
              FAILED вҶҗв”Җв”ҳ
                вҶ“
            QUEUED (retry)
```

| status | description |
|--------|-------------|
| `QUEUED` | task created, waiting for available worker |
| `ASSIGNED` | worker selected, preparing workspace |
| `RUNNING` | pipeline executing (current stage tracked) |
| `SUSPENDED` | worker offline, task may still be executing вҖ” awaiting reconnection |
| `COMPLETED` | all stages finished, code committed |
| `FAILED` | execution error вҖ” can be retried |
| `CANCELLED` | manually cancelled by user |

## machine selection

the dispatcher selects target machines using these criteria (in priority order):

1. user-specified machine via `--on` parameter
2. exclude offline and draining machines
3. filter by required capabilities (e.g., `claude`, `cursor`)
4. exclude machines above load threshold (default 85% cpu/memory)
5. exclude machines with all slots full
6. prefer machines that already have the project's base repository
7. select by lowest composite load score
8. tie-break by raw hardware capacity

## security model

- **authentication**: jwt tokens (access + refresh) with totp 2fa
- **authorization**: role-based access control (developer, lead, admin)
- **api tokens**: scoped personal access tokens for cli and api usage
- **worker auth**: one-time enrollment tokens + jwt for ongoing communication
- **audit trail**: all administrative actions logged to audit_logs table
